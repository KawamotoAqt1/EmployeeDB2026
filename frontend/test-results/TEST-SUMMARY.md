## テスト実施フロー

### フェーズ1: 環境準備 (5-10分)

```
1. バックエンド起動
   cd backend && npm run start:dev

2. フロントエンド起動
   cd frontend && npm run dev

3. テストデータ作成
   psql -U postgres -d employeedb -f test-data-setup.sql

4. 複数権限のユーザーでログイン可能な状態を確認
```

### フェーズ2: テスト実施

**完全テスト** (2-3時間)
- `company-management-tests.md` を使用
- 全106テストケースを実施
- 実施タイミング: 初回リリース前、大規模修正後

**クイックテスト** (40-50分)
- `quick-test-guide.md` を使用
- 主要機能のみ確認
- 実施タイミング: 小規模修正後、日常的な確認

### フェーズ3: 結果記録

```
1. test-execution-log.md に結果を記録
2. 発見したバグをバグレポートに記載
3. スクリーンショットを screenshots/ に保存
4. 総評を記入
```

### フェーズ4: 後処理

```
1. バグ修正
2. 再テスト
3. テストデータ削除(必要に応じて)
4. ドキュメント更新
```

## 重点テスト項目

### 高優先度

1. **企業登録・編集の基本動作**
   - 必須項目(企業名)のみでの登録
   - 拠点・担当窓口の追加
   - バリデーションエラーの表示

2. **検索・フィルタ機能**
   - 企業名検索
   - ステータス絞り込み
   - 検索結果の正確性

3. **権限制御**
   - VIEWER権限での編集不可
   - EDITOR/ADMIN権限での編集可能

4. **部署階層表示**
   - ツリー構造の正確な表示
   - 親子関係の視覚化

5. **削除確認モーダル**
   - 警告メッセージ表示
   - キャンセル・削除の動作

### 中優先度

1. ページネーション動作
2. レスポンシブデザイン
3. エラーハンドリング
4. URLバリデーション
5. メールバリデーション

### 低優先度

1. パフォーマンス測定
2. アクセシビリティ詳細確認
3. 大量データでの動作

## 既知の注意事項

### テスト実施上の注意

1. **権限テスト**
   - 複数の権限レベルのユーザーアカウントが必要
   - ADMIN, EDITOR, VIEWER の3種類を事前に準備

2. **ページネーションテスト**
   - 21件以上のデータが必要
   - test-data-setup.sql で24件作成される

3. **部署階層テスト**
   - テスト企業1(test-company-001)が最適
   - 4階層の部署構造を含む

4. **案件連携テスト**
   - Phase 3実装後に実施可能
   - 現時点では案件データが必要

### 技術的制約

1. **フロントエンドのみのテスト**
   - バックエンドAPIの詳細テストは別途必要
   - APIエンドポイントのテストは api-test-script.md を参照

2. **自動テストなし**
   - 現在はすべて手動テスト
   - 将来的にE2Eテスト(Playwright等)の導入を推奨

3. **クロスブラウザテスト**
   - Chrome, Edge, Firefoxでの確認を推奨
   - IEは非サポート

## 付録

### A. テストケース番号の読み方

**例**: T1-2-3
- T: Test(テスト)
- 1: カテゴリ番号(1=表示系、2=登録系、...)
- 2: サブカテゴリ番号
- 3: テストケース番号

### B. 重要度の定義

- **致命的**: システムが動作しない、データ損失の可能性
- **重大**: 主要機能が使用できない
- **中程度**: 機能に影響があるが回避策がある
- **軽微**: 見た目の問題、小さな不具合
